# 静态预警策略的工程演进：从 map_local + kd-tree 到 deskew + 暴力遍历

## 1 引言

在塔吊近场安全监控系统中，静态预警模块负责实时计算被监控对象（如标准节、障碍物）与周围静态环境之间的净空距离。该模块的核心设计决策之一是**静态环境数据源的选取**以及与之配套的**邻域搜索策略**。本章记录了整个静态预警方案从初始设计到最终落地的完整演进过程，包括：

1. 初始方案采用 LIO-SAM 局部地图（`map_local`）作为静态环境表征，配合 kd-tree 进行邻域搜索；
2. 实际测试中发现 **"伪静态点"** 问题——被监控对象自身的历史点云被固化在局部地图中，导致静态净空计算失真；
3. 最终切换为**去畸变点云（deskew）**作为静态环境输入，并采用**暴力遍历**策略，在保证判别准确性的同时满足实时性要求。

---

## 2 初始方案：map_local + kd-tree

### 2.1 设计初衷

LIO-SAM 实时维护的局部地图 `/lio_sam/mapping/map_local` 是一个经过多帧累积、去重、降采样后的稠密点云，具有以下优势：

- **覆盖范围广**：包含历史多帧观测，环境表征完整；
- **噪声低**：经过体素滤波和动态点剔除，点云质量较高；
- **坐标统一**：统一在 `map` 坐标系下，便于与视觉对象的位姿直接对比。

基于上述优势，初始方案选择 `map_local` 作为静态环境输入。考虑到局部地图规模通常可达 $10^5$ 量级，邻域搜索采用 kd-tree（`pcl::KdTreeFLANN`），期望将最近邻搜索的复杂度从 $O(N)$ 降至 $O(\log N + M)$。

### 2.2 实现架构

初始方案的数据流如下：

```
/lio_sam/mapping/map_local
    ↓
static_cloud_self_filter  [history_duration_sec = 3.0s]
    ↓  （过滤对象自身及其历史运动轨迹上的点）
/warning/static_cloud_filtered
    ↓
warningEvaluator
    ↓  kd-tree radiusSearch
静态净空距离 + 预警分级
```

其中 `static_cloud_self_filter` 节点维护了一个 `history_spheres_` 队列，记录被监控对象在最近 3 秒内的历史位姿，并以球体扫掠体（swept segment）为掩模，剔除落入对象自身运动轨迹范围内的点云。

### 2.3 遇到的问题：伪静态点

在实车/实机测试中发现，即使经过 `static_cloud_self_filter` 过滤，`warningEvaluator` 计算出的静态净空距离仍然异常偏小，频繁触发误报警。深入排查后发现问题根源：

**"伪静态点"（Pseudo-Static Points）**——被监控对象（如塔吊吊起的标准节）在 map 坐标系下的历史点云已被 LIO-SAM 的建图过程固化为局部地图的一部分。虽然 `static_cloud_self_filter` 能根据对象的历史位姿进行过滤，但存在以下局限：

1. **位姿对齐误差**：map_local 中某一点的坐标是历史时刻传感器观测经 SLAM 优化后的结果，而 `static_cloud_self_filter` 中对象的历史位姿来自视觉分割模块的外推估计，两者之间存在帧间偏差；
2. **地图更新滞后**：LIO-SAM 的局部地图更新存在时间窗，对象移动后其历史点云不会立即从地图中移除；
3. **多帧累积效应**：标准节等大块物体在地图中留下的"痕迹"宽于单帧点云，简单的球体/圆柱体掩模难以完全覆盖。

上述因素导致对象周围始终残留大量"看起来是静态障碍物、实际上是被监控对象自己"的伪静态点，使得静态净空距离被严重低估。

## 3 方案切换：deskew 点云 + 暴力遍历

### 3.1 切换为 deskew 点云的动机

与 `map_local` 不同，**去畸变点云（deskewed cloud）** `/lio_sam/deskew/cloud_deskewed` 是单帧（或极短滑动窗口）的原始观测，具有以下关键优势：

1. **无历史累积**：不包含对象在 map 中的历史"痕迹"，从根本上消除了伪静态点；
2. **时间同步性强**：deskew 点云的时间戳与当前视觉分割结果几乎同步，位姿对齐精度更高；
3. **自身过滤更干净**：`static_cloud_self_filter` 只需基于当前对象位姿做单帧过滤，无需维护长历史队列。

因此，系统将静态环境数据源从 `map_local` 切换为 `deskew`，并在 launch 文件中通过参数控制：

```xml
<arg name="warning_static_cloud_source" default="deskew" />
```

当 `static_cloud_source == "deskew"` 时，`static_cloud_self_filter` 的输入话题自动指向 `/lio_sam/deskew/cloud_deskewed`，同时 `history_duration_sec` 缩短为 $0.2\,\text{s}$（仅保留极短历史用于运动轨迹扫掠过滤）。

### 3.2 搜索策略的配套调整

切换为 deskew 后，静态点云规模发生显著变化。实测数据表明：

- **map_local 模式**：过滤后的点云规模可达 $10^5$ 量级（$10^5 \sim 1.7 \times 10^5$）；
- **deskew 模式**：单帧去畸变点云经过自身过滤后，规模仅为 $10^4$ 量级（$1 \times 10^4 \sim 2.5 \times 10^4$）。

由于点云规模下降了一个数量级，kd-tree 的算法优势被其常数开销（递归剪枝、非连续内存访问、动态内存分配）所掩盖。为验证两种搜索策略在 deskew 规模下的实际表现，系统内置了对比 benchmark：

| 点云规模 | 暴力遍历 | kd-tree | 加速比 |
|:---|:---|:---|:---:|
| 10k | ~0.06 ms | ~0.01~5.5 ms（剧烈波动）| **< 1x（更慢）** |
| 20k | ~0.13 ms | ~0.02~5.3 ms（剧烈波动）| **< 1x（更慢）** |
| 100k（map_local 参考）| ~0.54 ms | ~0.06 ms | **~9x** |
| 170k（map_local 参考）| ~0.89 ms | ~0.03 ms | **~28x** |

**分析**：kd-tree 的 `radiusSearch` 每次调用都需要向 `std::vector<int>` 和 `std::vector<float>` 动态插入返回点。当搜索半径内点数不稳定时，vector 反复 `malloc/realloc` 的内存分配开销成为主导因素，导致 kd-tree 耗时波动极大（0.01 ms ~ 5.5 ms）。相比之下，暴力遍历是纯粹的线性内存扫描，无堆分配、cache 友好、稳定性极高。

**结论**：在 deskew 的 $10^4$ 量级输入下，暴力遍历的平均耗时已处于亚毫秒级（0.06~0.14 ms），完全满足 10 Hz 评估频率的实时性要求，且稳定性显著优于 kd-tree。

### 3.3 最终方案架构

综合以上分析，系统最终采用的静态预警架构为：

```
/lio_sam/deskew/cloud_deskewed
    ↓
static_cloud_self_filter  [history_duration_sec = 0.2s]
    ↓  （基于当前对象位姿做单帧自身过滤）
/warning/static_cloud_filtered
    ↓
warningEvaluator
    ↓  暴力遍历（逐点计算欧氏距离）
静态净空距离 + 预警分级
```

该架构的优势在于：
- **准确性**：deskew 点云无历史累积，配合短历史自身过滤，彻底消除了伪静态点；
- **实时性**：$10^4$ 量级下的暴力遍历耗时低于 0.15 ms，10 Hz 评估周期内可忽略；
- **简洁性**：无需维护 kd-tree 索引，代码逻辑直观，调试成本低。

---

## 4 讨论：若未来规模扩大怎么办？

尽管当前 deskew + 暴力遍历是最优解，但系统仍保留了向大规模地图扩展的能力：

1. **kd-tree 代码接口已保留**：`warningEvaluator` 中仍维护 `static_kdtree_`，并在点云回调中自动构建索引；
2. **数据源可一键切换**：只需修改 launch 参数 `warning_static_cloud_source` 为 `map_local`，即可切换回局部地图模式；
3. **性能储备**：实测表明，当输入规模超过 $10^5$ 时，kd-tree 可展现出 5~28 倍的加速比，足以应对未来更稠密地图或更高评估频率的需求。

因此，当前方案是一个**面向当前数据特征的务实选择**，同时为未来的规模扩展预留了清晰的升级路径。

---

## 5 总结

本章完整记录了静态预警策略的工程演进：

| 阶段 | 数据源 | 搜索策略 | 遇到的问题 | 解决方式 |
|:---|:---|:---|:---|:---|
| 初始设计 | `map_local` | kd-tree | 伪静态点导致净空计算失真 | 切换数据源 |
| 最终方案 | `deskew` | 暴力遍历 | 无 | — |

**核心教训**：空间搜索算法的选型不能仅依赖理论复杂度，必须结合实际数据规模、内存分配特征和具体业务场景（如"自身过滤"的精度要求）进行综合评估。在本系统中，deskew 点云的小规模、高时效性特征使得暴力遍历成为比 kd-tree 更优的工程选择。
